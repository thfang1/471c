## Minor Extension: Common Subexpression Elimination and Copy Propagation

## 1. Justification

The current optimization loop performs constant propagation, constant folding, and dead code elimination. These passes reduce constant expressions and eliminate dead bindings, but do not address two related sources of redundancy: (1) the same pure expression computed more than once, and (2) variable-to-variable copy bindings that add indirection without adding information.

These two optimizations are proposed together because they form a natural pipeline: CSE introduces copy bindings as a side effect of deduplication, and copy propagation then eliminates exactly those bindings. Together they clean up a significant class of IR redundancy, especially in code generated by CPS conversion and other compiler passes.

**Who benefits:**
- Users writing numeric or data-intensive code that repeats subcomputations benefit from reduced execution cost.
- The optimization loop becomes more powerful: CSE and copy propagation create new opportunities for dead code elimination, and the combined effect is greater than either pass alone.
- Downstream passes benefit from fewer live variables and a smaller IR.

## 2. User-Facing Behavior

Both passes are transparent to the user. Consider a program with a repeated subexpression:

```scheme
(let ([a (* x x)])
  (let ([b (* x x)])
    (+ a b)))
```

**After CSE:**
```scheme
(let ([cse0 (* x x)])
  (let ([a cse0])
    (let ([b cse0])
      (+ a b))))
```

**After Copy Propagation:**
```scheme
(let ([cse0 (* x x)])
  (let ([a cse0])
    (let ([b cse0])
      (+ cse0 cse0))))
```

**After DCE (existing):**
```scheme
(let ([cse0 (* x x)])
  (+ cse0 cse0))
```

The multiplication is computed only once. The user sees only the correct result, computed more efficiently.

## 3. Pipeline Impact

**CSE** is added as a new pass in the optimization loop, running after constant folding and before dead code elimination. It maintains a table of seen pure expressions (keyed by structural equality) as it traverses the binding chain. When a duplicate is found, it introduces a new shared binding and replaces subsequent occurrences with a reference to it. Only pure, side-effect-free expressions are candidates.

**Copy Propagation** is implemented as an extension of the existing constant propagation pass. The current pass maintains an environment mapping variable names to known values (immediates or references). The extension adds one case: if a variable is bound to another variable (the RHS of a let is a plain variable reference), substitute that variable at all downstream use sites.

Both passes integrate naturally into the existing while-changes optimization loop without additional scheduling complexity. Running CSE before copy propagation, and copy propagation before DCE, forms the optimal ordering.

## 4. Success Criteria

- **Correctness:** All existing test cases pass unchanged.
- **CSE completeness:** In programs with known duplicate pure subexpressions, CSE eliminates all syntactically identical occurrences within a binding scope.
- **No false positives:** Expressions with side effects are never merged by CSE.
- **Copy propagation completeness:** After one pass, no let-binding of the form `(let ([x y]) ...)` where `y` is a plain in-scope variable remains in the output.
- **Synergy:** Combined with DCE, programs with redundant computations show a measurable reduction in the number of let-bindings in the output IR compared to running neither pass.
